# 🦊 Pika Review: Architectural Rules Example

This file defines the project-specific constraints that the AI Sentinel will enforce during every scan. 
Rename this file to `.pika-rules.md` in your project root to activate it.

---

## 🏛️ General Architecture
- **State Management**: We use `Zustand` for global state. Avoid `Redux` or `Context API` unless explicitly required for library compatibility.
- **Data Fetching**: All API calls must use the `src/services/apiClient.ts` wrapper to ensure consistent error handling and logging.
- **Component Structure**: Prefer **Functional Components** with Hooks over Class Components.

## 🔐 Security Standards
- **Sensitive Data**: Never log raw PII (Personally Identifiable Information) or API keys.
- **Sanitization**: All user-provided HTML must be sanitized using `dompurify` before rendering.
- **Auth**: Use the `@useAuth` hook for checking permissions; do not read tokens directly from `localStorage`.

## 💅 Styling & UX
- **Design System**: Use variables from `src/theme/tokens.js` for colors and spacing. Do not use hardcoded hex codes.
- **Accessibility**: Every interactive element (`<button>`, `<a>`) must have an `aria-label` or descriptive text.
- **Performance**: Large lists (over 50 items) must be virtualized using `react-window`.

## 📦 Dependency Management
- **Imports**: Use absolute imports (e.g., `@/components/...`) instead of relative paths (`../../../components/...`).
- **Utilities**: Check `src/utils/` before adding a new utility library like `lodash`.
